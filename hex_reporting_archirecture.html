Hexagonal Architecture

Document Generation
Microservice Design

A comprehensive architecture blueprint for building scalable, maintainable document generation services using Ports and Adapters pattern.

Executive Summary

This architecture document outlines a robust microservice design for document generation using Hexagonal Architecture principles.

  • Multi-channel Input: REST API, Kafka messaging, and scheduled tasks
  • Core Decoupling: Business logic isolated from infrastructure concerns
  • Extensible Design: Framework-agnostic core with pluggable adapters
  • Production Ready: Containerized deployment with observability

1. Overview & Goals

1.1 System Responsibilities

The Document Generation Microservice orchestrates complex workflows involving data retrieval, template processing, document rendering, and storage. It supports multiple entry points including synchronous REST API calls, asynchronous Kafka messages, and scheduled batch jobs.

Real-world Example: A legal document generator creating contracts from user forms, sending for e-signature via DocuSign integration, and archiving signed copies.

1.2 Core Architectural Goals

Modularity & Decoupling

Strict separation between business logic and infrastructure using Hexagonal Architecture principles. Changes in one component have minimal impact on others.

Framework Agnosticism

Core logic is pure Java with no framework dependencies. Enables easy migration between Spring Boot, Quarkus, or Micronaut.

Testability & Maintainability

Isolated core logic enables comprehensive unit testing with mocked ports. Clean separation makes the system more maintainable and reduces bug propagation.

Scalability & Resilience

Asynchronous processing via Kafka enables horizontal scaling. Fault isolation ensures partial failures don't crash the entire system.

2. Hexagonal Architecture Principles

2.1 Core Tenets of Ports and Adapters

The Hexagonal Model

Core
Domain
Inbound
Adapters
Outbound
Adapters
Ports as Contracts
Inbound Flow

External actors → Inbound Adapters → Ports → Core Use Cases

Core Logic

Pure business logic with no external dependencies

Outbound Flow

Core → Ports → Outbound Adapters → External Systems

Hexagonal Architecture structures applications around a central core, visualized as a hexagon, containing domain and application logic. This core communicates with the outside world through well-defined interfaces called "ports," with implementations called "adapters."

The Dependency Rule

All dependencies must point inward, toward the core domain. The core has no knowledge of the outside world - it doesn't depend on frameworks, databases, or UI technologies.

Inbound adapters depend on core ports
Outbound adapters implement core ports

2.2 Comparison with Traditional Layered Architecture

Feature Traditional Layered Hexagonal
Dependency Direction Top-down, unidirectional Inward-pointing
Coupling High coupling between layers Low coupling
Testability Difficult to test in isolation High testability with mocks
Flexibility Low - changes require layer updates High - swap adapters easily
Logic Leakage Prone to infrastructure mixing Prevents leakage

7. Testability Approach

7.1 Testing the Core Domain

Unit Testing Use Cases

Test use cases with mocked ports to verify orchestration logic in complete isolation.

@Test
void generateReport_success() {
when(dataPort.fetchData()).thenReturn(testData);
when(templatePort.applyTemplate()).thenReturn(content);

Result result = useCase.generateReport(command);

verify(storagePort).storeDocument();
assertThat(result).isSuccess();
}

Domain Logic Testing

Domain objects are pure POJOs - trivial to instantiate and test without any dependencies or mocks.

7.2 Testing Adapters

Adapter Unit Tests

Test data transformation and error handling logic within adapters.

  • • Mock external clients (S3, Kafka, etc.)
  • • Verify proper exception translation
  • • Test data mapping correctness

Integration Tests

Test with real external systems using Testcontainers for disposable environments.

  • • Real PostgreSQL database
  • • Actual Kafka broker
  • • S3-compatible storage

7.3 Enforcing Architectural Constraints

ArchUnit Validation

Use ArchUnit to write unit tests that verify architectural rules and prevent dependency violations.

Dependency Rules
// Core must not depend on frameworks
noClasses().inDomain()
.should().dependOnClassesThat()
.resideInAnyPackage(
"org.springframework.."
)
Layer Access Rules
// Adapters must not call other adapters
classes().that().resideInAPackage("..adapter..")
.should().onlyAccessClassesThat()
.resideInAnyPackage(
"..port..", "..domain.."
)
